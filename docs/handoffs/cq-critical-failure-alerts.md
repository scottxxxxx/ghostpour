**Critical failure alerting handoff for the CQ team**
*From: GhostPour (Scott + Claude). Date: 2026-05-20.*

Hey CQ team, sharing how we built the critical failure alerting on our side in case it's a useful template for something similar over there. The motivation was the silent failure incident a couple weeks back, two weeks of broken capture before anyone noticed, and the obvious fix was to surface system level failures to a human via email. We shipped this last week and it's now live end to end with on call email landing in my inbox.

The shape is three things stacked. A recipients table that holds the operator addresses, with an optional list of category subscriptions per recipient so different people can subscribe to different alert types, or leave it empty to receive everything. A second table tracks open incidents, one row per unique fingerprint where the fingerprint is category plus subject string (so cq_unreachable:cq or provider_auth_failed:openai are separate rows), while the incident is open, and the same row records the resolved state once the incident auto closes. And a service module wraps both, exposes a single report_incident function for callers to call at failure sites, handles the dedup, manages the auto resolve, and dispatches the email through Resend.

The suppression model matters and was the part we thought about most. Once an incident is open for a given fingerprint, repeat reports just bump a trigger count and the last seen timestamp, nothing new goes out by email, that's the once per incident rule. Auto resolve happens by time, not by an explicit "I'm better now" hook, if the fingerprint goes quiet for 30 minutes the row gets marked resolved, and the next failure with the same fingerprint opens a fresh row and triggers a new email. The 30 minute window is tunable in one place. We chose time based auto resolve specifically because detecting "I'm fixed now" is harder than just watching whether the symptom recurs, and it avoids needing an ack UI for v1.

There's an admin dashboard tab where you add and remove recipients, browse the incident history (open rows in red, resolved in green, trigger count plus emailed count), plus a test send button that fires a synthetic incident under whichever category you pick so you can verify deliverability without waiting for a real failure. The test sends land in the history table the same as any real incident so you can confirm the email path actually went through.

A few things turned out to be load bearing. The service swallows all downstream failures from Resend, network blips, anything, alerting can never break the request that triggered it because that would compound the user pain. The category catalog is a single source of truth, the same dict drives the dashboard picker, the subscription filter, and the wire site callers, so adding a new category is a one line change. And the from address has to be on a Resend verified domain or the send returns a no_api_key style skip, that one cost us a few minutes when we first tried.

Implementation surfaces if you're scoping the work: a schema migration adding the two tables with a partial unique index on fingerprint where resolved_at is null for cheap dedup, a service module around 300 lines with the report_incident function plus a list_incidents helper for the dashboard, admin endpoints for the recipient CRUD plus a test send route under your admin namespace, a dashboard tab with the add form and history view, and the wire ins at whichever critical failure points matter for CQ. We wired one category fully and left two more defined in schema and UI but not yet hooked into wire sites since they need a bit more context threading at the call points. The schema accommodates them when you're ready, partial scope is fine.

Skip in v1: ack mechanism (the timer is enough), per recipient delivery channels (we did email only, Slack and PagerDuty are extensions on top), and per category subscription frequency rules (digest mode, etc). All sit cleanly on top of the same core.

For Resend setup specifically, you need a verified sender domain (DKIM and SPF DNS records) and the API key plus webhook secret. The webhook secret is only needed if you want bounce and complaint webhooks to auto populate a suppression list, you can skip it initially. We put both secrets in GCP Secret Manager with a cross project IAM grant on the running service account. The verified domain restriction surprised us, the API key alone isn't enough, sending from an unverified domain falls back to from@resend.dev and their docs warn against it.

On Resend domain limits while you're scoping: Resend supports multiple verified domains per account, no restriction on which domain names you pick (you just need DNS control). Subdomains are first class. Exact count per tier shifts with their pricing, free tier is one domain, paid tiers go up from there, worth glancing at their dashboard limits page before committing. The CQ alerts can run on a different domain than ours, doesn't have to share.

Total scope on our side was around a thousand lines of code and 22 tests, shipped in one PR. The reference PR on our side is cloudzap PR #194 if you want to read the diff. Happy to walk through any of it in more detail, or pair on the schema if it would help.
