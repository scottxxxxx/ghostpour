**Image attachment incident postmortem (cross stack: GP + SS)**
*Incident date: 2026-05-20. Postmortem written: 2026-05-21. Authors: GP team (Scott + Claude), with diagnoses contributed by SS team.*

Scott was using SS Studio on iPhone the morning of 2026-05-20. He attached an image to a chat in Ask mode and asked about its contents. The AI gave vague answers across three turns, eventually hallucinating a "watermark" that wasn't in the image. Scott took a screenshot at 11:45 AM Central, flagged it to GP as a quality bug, and we kicked off triage that spanned both teams across the rest of the day. The story turned out to involve two unrelated bugs, one on each side of the wire, neither of which was the thing we first thought it was. Both have shipped fixes. This is the writeup.

The first thing we found in GP's usage_log was two image bearing requests for Scott's user_id at 14:21 and 14:22 UTC (9:21 and 9:22 AM Central). Both came through with `prompt_mode: "What Are We Missing?"` despite the SS UI in the screenshot clearly showing the Ask pill highlighted. That looked like the smoking gun. We pointed SS at it, hypothesis being that the iOS app was sending the wrong `prompt_mode` value on the wire, maybe sticky state from a prior quick prompt tap. SS walked the V1 SessionView code path and found a real bug right where I was looking: a `lastUsedMode` state var that got written every time someone executed a query or picked a mode from the carousel. Three variants of the main quick action button all defaulted to it, so if `lastUsedMode` was set and wasn't "Freeform", the next tap of that button fired with that mode and its system prompt, no confirmation. Long press into the carousel, tap "What Are We Missing?", that locked WAWM as the silent default until the next deliberate pick. SS shipped a fix the same day: all three V1 button variants now always fire `promptModeStore.modes.first` regardless of history.

While walking the code, SS noted that V2 Studio (which is the default per `useStudioLayout` and the layout Scott is targeting going forward as V1 is deprecated) didn't have the same sticky pattern. `QueryComposerSheet` builds a fresh `PromptMode` per send. If Scott was on Studio at the time of the incident, the V1 fix wouldn't cover it, and we'd need a different explanation. SS asked us to check the `user_query` slot in the wire payload on those two 14:21/14:22 calls. Their reasoning: the freeform Ask path always populates `user_query`, the quick prompt tile path passes nil. Empty `user_query` means the user tapped the WAWM tile, populated `user_query` means there's a code path SS hadn't found.

I pulled the raw_request metadata on both calls. Both had empty `user_query`. The only user message content was the assembled meeting context plus an `[1 image(s) attached for visual context]` marker, no Scott typed text. So those two calls were tile taps, confirmed. SS's tile tap diagnosis was the right one for the data we were looking at.

Here's where the investigation got interesting. While reviewing the wire data, I noticed those two calls were timestamped at 14:21 and 14:22 UTC, which is 9:21 and 9:22 AM Central. Scott's screenshot showed message timestamps of 11:39 to 11:41 AM Central (16:39 to 16:41 UTC). Two hours apart. The 14:21/14:22 calls were happening during a live ABM meeting (the project context block in the request confirmed it, "Recent transcript ~2 minutes" full of in meeting chatter). The screenshot conversation was a post meeting review of a different image, two hours later. I had been treating them as one incident when they were two.

I checked the GP usage_log for the 16:30 to 16:55 UTC window across every user. Zero rows. The screenshot conversation never reached GP at all. That conversation routed through a path that bypasses our server entirely. Two such paths exist: on device Foundation Models (the `onDevice/foundation-models` provider, runs locally on the device with no network call) or OpenRouter direct (BYOK, iOS straight to OR with the user's key, GP not in the loop). The vague answers and hallucinated watermark in the screenshot fit the on device profile better, that's typical confabulation from a less capable local model.

SS confirmed this on their side. There was a relaunch demote bug in `LLMService` where `providerIsUsable("cloudzap")` was reading the async populated `isSignedIn` flag from `CloudZapAuthManager` before `checkExistingSession` had restored the JWT, so signed in Studio users would silently fall back to on device every launch despite having a valid JWT in Keychain. SS shipped a fix the same day, mirroring the BYOK Keychain direct pattern. So the screenshot conversation ran on Apple Foundation Models because the cloudzap provider check raced against auth state restoration on app launch.

To recap the actual root causes: SS V1 had a sticky `lastUsedMode` that silently changed the active prompt mode on quick action button tap (real but pre existing, found while investigating). SS V2 had an auth restore race condition that demoted signed in users to on device Foundation Models on every launch (the actual cause of Scott's screenshot incident). Both are fixed. The 14:21/14:22 tile tap calls weren't bugs at all, they were intentional taps that Scott made during the live meeting, with the WAWM tile selected and an image attached, no typed query. The empty `user_query` and the prompt mode mismatch with the screenshot UI were both honest reflections of what was happening at that time.

On the GP side, we shipped PR #195 as a defensive measure during the early phase of triage. It adds an image acknowledgement rule to `defaultUserInstructions` in protected-prompts.json, telling the model to describe attached image content before applying any other framing. It's still in prod and applies to any future case where a GP routed call has a prompt mode framing that doesn't fit an attached image. It does not apply to the actual screenshot conversation since on device Foundation Models never see our system prompts, but the defensive value is real and we left it in. We also have a follow up branch ready that adds a similar escape hatch to each individual quickPrompt's `systemPrompt`, but given the root causes were elsewhere we agreed with SS to hold that branch rather than land it. Easy to ship later if we ever see another case where the prompt design itself is the issue.

Two notes on what could have gone better. First, the GP team (specifically me) conflated two incidents that happened to have a user, an image, and a confusing AI response in common. The 9:21 AM tile taps and the 11:39 AM screenshot conversation were two different sessions on two different paths. Cross referencing the user's screenshot timestamps against the wire telemetry timestamps before forming a diagnosis would have caught that immediately. I'm leaving the lesson here for the next cross team triage: when a user reports a model quality issue, check first whether the call actually reached your server in the reported window. Zero telemetry is itself a signal, not noise. Second, the path SS spent time investigating (the V2 QueryComposerSheet mode handling) turned out to be unrelated to the screenshot incident. SS did clean work and found a real V1 bug along the way, but the investigation was steered by my misframing. They were patient about it and the V2 auth restore catch came out of their broader review rather than the wrong lead specifically.

Total scope shipped from this incident: SS shipped two iOS fixes (V1 `lastUsedMode` silent default, V2 auth restore race condition). GP shipped one config bump (PR #195, image acknowledgement rule). One follow up branch is parked (per quickPrompt escape hatch). Both incident threads are closed.

SS deserves explicit credit on this. Their tile tap hypothesis was the right read on the data we could see. The V1 SessionView walk was thorough and the fix was clean. The independent catch of the V2 auth restore race condition turned out to be the actual root cause of the user visible bug, found by following their own broader review rather than the (wrong) lead I'd pointed them at. Both fixes shipped same day. The collaboration pattern of careful code walking plus tight feedback loops via async messages worked here, would repeat.
