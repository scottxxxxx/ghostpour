### Added
- **Per-app version metadata endpoint `GET /v1/app/version`** — SS asked for a "you're on an old version" toast wired through the existing remote-config pipeline. Their proposed shape baked in single-tenant assumptions (fields prefixed `ios_`, no app identifier on the request, App Store URL hardcoded server-side), which would have meant a breaking wire rev the day a second app joins the gateway. Built it multi-tenant from day one instead: dedicated endpoint, `X-App-Bundle-Id` header required, response has a `platforms` block so Android/macOS/etc. don't need a contract change later. Backed by `config/app-versions.yml` keyed by bundle id with `platforms.<platform>.{latest_version, min_supported_version, upgrade_url}`. Adding a new app is one YAML row, no code. Unknown bundle id returns 404 (gateway doesn't know that app), missing header returns 400 (request shape problem), entry without a `platforms` block also returns 404 (surface misconfiguration immediately rather than silently returning empty data). No auth — call fires pre-login on app launch. New `app/services/app_version.py` (loader + lookup), `app/routers/app_version.py` (endpoint), Settings field `app_versions_path` defaulting to `config/app-versions.yml`. Seed entry for SS bundle `com.weirtech.shouldersurf` with `latest_version: "1.11"` (current observed in telemetry) and a TestFlight placeholder URL pending the real one from SS. 12 new tests covering all four endpoint paths (missing/empty/unknown/known bundle id), no-auth requirement, cache-control header, and four loader edge cases (missing file, malformed YAML, non-mapping root, real shape). Full suite 675 pass, no regressions.
- **Cert pin auto-republish daemon + dashboard expiry banner** — closes the calendar-reminder loop on top of PR #207. A lifespan-spawned daily task reads the latest manifest's `expires_at`; when within 7 days, fetches the live TLS chain off `cert_pin_self_host` (new Settings field, default `cz.shouldersurf.com`), takes the intermediate + both roots (skips the leaf so we don't follow the 60-day LE renewal cadence), and publishes a fresh manifest. Pin-selection policy: dynamic from live chain — if LE rotates an intermediate, auto-republish ships the new pins on its own. Any pin-set change vs. the previous publish fires a `cert_pin_auto_republish` incident (new alerting category) with the prior + new pin lists so the operator sees it, but the new manifest still gets published (per the chosen policy, hands-free property is preserved). New helper `fetch_current_chain_pins(host)` in `app/services/cert_pin_signing.py` uses Python 3.13+'s `SSLSocket.get_unverified_chain()` so we don't need an openssl subprocess. New admin endpoint `GET /webhooks/admin/cert-pins/status` returns the banner data dict (level: green/yellow/red, text, version, expires_at, days_remaining, last_check). New dashboard banner above the tabs shows yellow at 14d remaining, switches to "auto-republish next tick" wording at 7d, and goes red on auto-republish failure, expired manifest, missing signing key, or no manifest published yet. Green level suppresses the banner so it only takes screen space when there's something to see. Auto-published rows record `created_by_admin_key_suffix = "auto-republish"` so audit queries can tell automated from manual publishes. Tests: 13 new (status banner across all level transitions, no-signing/healthy/in-window/pin-change branches of `maybe_auto_republish`, alerting hook fires on pin change, admin status endpoint round-trip, live integration test against prod chain). Total cert pin signing tests now 24, full suite 664 pass.
- **Signed cert pin manifests at `GET /v1/config/cert-pins` (proposal from SS at `ShoulderSurf/docs/CERT_PINNING_PROPOSAL.md`)** — when Let's Encrypt rotates the cert chain on `cz.shouldersurf.com`, every existing iOS install with stale baked-in pins fails at TLS handshake until the user updates the app. This PR decouples pin updates from app releases by serving a small Ed25519-signed JSON manifest (`{version, pins, issued_at, expires_at, signature, algorithm}`); iOS verifies the signature against a baked-in public key so trust comes from the signature, not the transport (no circularity — the pin manifest fetch itself goes over plain HTTPS, but a MITM cannot forge a valid signature without the private key). Migration v24 adds `cert_pin_manifest` with monotonic `version PRIMARY KEY` for rollback protection. New service `app/services/cert_pin_signing.py` handles key load, canonical JSON serialization (sorted keys, no whitespace — must match the iOS-side canonicalization byte-for-byte), Ed25519 sign + verify. Public read endpoint in `app/routers/cert_pins.py`, registered ahead of the config router so `/v1/config/cert-pins` wins over the `/v1/config/{name}` catch-all. Admin endpoints in `app/routers/webhooks.py`: `POST /admin/cert-pins/publish` (signs + persists a new monotonic version) and `GET /admin/cert-pins/current` (returns the latest manifest + public key for handing to SS + a round-trip `verified` flag as a key-mismatch sanity check). Key custody: 32-byte raw Ed25519 private key, base64-encoded, lives in env var `CZ_CERT_PIN_SIGNING_KEY_RAW_B64` (mapped to GCP Secret Manager secret `cert-pin-signing-key-raw-b64`); generated locally on the operator machine with `openssl genpkey -algorithm Ed25519` and NEVER committed to the repo (`.gitignore` hardened with `private-local-secrets/` + `*signing_private*` patterns). 11 tests in `tests/test_cert_pin_signing.py` cover sign/verify round-trip, tampered payload rejection, missing/malformed/wrong-length key handling, public 404 → 200 transition, monotonic version bump across publishes, admin round-trip verification, admin-key gating, and empty-pins rejection.

### Changed
- **Captions cleanup primary swapped to DeepSeek V3.2-exp via OpenRouter, with Anthropic Haiku 4.5 as the fallback** — eval on the 2026-05-29 QA Testing transcript + ground truth found DeepSeek V3.2-exp matched Haiku 4.5's ROUGE-L (0.812 vs 0.815) and gap-closed (50.9% vs 51.5%) at roughly 1/8 the cost (~$0.028 vs $0.22 per projected 1hr meeting). Across the broader eval corpus (5+ runs) DeepSeek has shown latency variance (11-72s) and one quality crater (ROUGE 0.51), so we wrap the primary call in a 30-second timeout and fall back to Haiku 4.5 (direct via Anthropic) on timeout, exception, or empty response. Both failing returns None (existing silent-skip behavior). Haiku has tighter observed latency (6-16s) and quality variance, which makes it the right safety net. `config/providers.yml` adds a new `openrouter` provider entry (api_format: openai-compatible) so the routing layer can dispatch the primary; the fallback uses the existing `anthropic` provider. New constants in `app/services/transcript_cleanup.py`: `_PRIMARY_PROVIDER/_MODEL`, `_PRIMARY_TIMEOUT_SECS=30.0`, `_FALLBACK_PROVIDER/_MODEL`. Both routes log a single attribution line ("Transcript cleanup ok (primary)" or "(fallback)"). Tests extended to 21 (was 18) with three new fallback cases: primary error falls back, primary timeout falls back, primary empty falls back, both routes failing returns None, both routes returning empty returns None.

### Changed
- **Fix admin `/admin/users` gauge to derive from `usage_log`, not `monthly_used_usd`** — the dashboard's `hours_used` and `percent_used` fields were computed from `users.monthly_used_usd`, the budget-gate counter that `usage_tracker.record_cost` early-returns out of for unlimited tiers (`effective_limit == -1`). Consequence: every Plus/Pro/admin row showed `0.0 hrs` and `0%` regardless of actual activity (e.g., the project owner's own account: 1,232 lifetime calls, 198 distinct meetings, $12.58 spent — gauge said 0 hours). The same path also affected Free users when the counter drifted from reality (observed: a Free user with a single $0.0008 query showed `7 of 7` with a red bar, while the per-user detail panel correctly returned no recent activity). The SQL query in `/admin/users` already computes an honest `total_cost_usd` by summing `usage_log.estimated_cost_usd` over the dashboard's `days` window; the change is one local variable: `hours_used` and `percent_used` now derive from that window cost. `hours_limit` still derives from `monthly_cost_limit_usd` since that is the policy not the consumption. `monthly_used_usd` itself is left untouched — it keeps its current role as the budget-gate state, where staying at zero for unlimited tiers is the correct behavior. No schema change, no migration, no SS coordination. Validated by spot-checking the project owner's row (~$1.05 in last 7 days → ~5.5 hours derived) and the previously-suspicious Free user (~$0.0008 in window → 0 hours, 0.2% — replaces the false `7 of 7`).
- **Fix action-item over-attribution in the report prompt** — the single rule "Attribute action items to specific people by name when the transcript supports it" (`app/services/meeting_report.py`) was producing two failure modes: (1) soft suggestions ("a great step would be us taking a look", "we should consider") got promoted to firm action items, and (2) owners were assigned by name-proximity rather than by who actually committed — e.g. a meeting where someone noted "Scott drafted the success criteria" produced a false "Scott: review the success criteria" action even though Scott never spoke to it and another person proposed the review. Replaced with two principles: an action requires a concrete commitment or assignment that someone took on (not a hypothetical), and the owner is whoever committed to doing it (not whoever is named nearby or authored a related document); unowned proposals become open questions. Validated with before/after report generation against two real meetings: in the first, the false "Scott" action correctly moved to an open question; in the second, Scott's genuine verbal commitment ("Yeah, I can talk to them") was correctly retained while borderline non-commitments dropped. No real actions were suppressed in either — dropped items were either genuinely ownerless or got consolidated. Single-line prompt change, easy to revert. No wire shape change, no migration, no SS coordination.
- **Tighten `suggested_tags` rule in the report prompt** — `app/services/meeting_report.py` line 71 previously instructed the model to return 1-4 tags from the taxonomy, which the model treated as "produce 3-4 every time" because the upper bound anchored its default. Result: tags became monotonous (every meeting got "Action Items", "Decisions", "Follow-ups") and stopped carrying any distinguishing signal. New rule: at most 2 tags, applied only when they capture what makes THIS meeting distinct from a typical meeting, with most meetings producing 1 tag or 0. Prompt now explicitly says "do not apply a tag just because it could plausibly fit" and requires a reason explaining what makes the tag distinguishing. Single-line prompt change, easy to revert by restoring the previous wording. No wire shape change, no migration, no SS coordination required. Worth eyeballing for a few meetings post-deploy to confirm tags actually got more discriminating instead of meetings losing tags they should have had.

- **Request-log streaming mislabel (PR #166)** — `StreamingBypassMiddleware` decided "(streaming)" from `"stream":true` in the request body before the handler ran. Handlers that override streaming and return JSON (Project Chat being the canonical case) were mislabeled in both the summary log line and the dashboard live-log, with body shown as `"(streaming)"` instead of the actual JSON response. Now decided from the response `content-type` (`text/event-stream`) inside the `http.response.start` callback; the two log-path branches collapsed into one. 5 new ASGI-level tests cover JSON-response-to-stream-request not mislabeled (the bug), real SSE labeled correctly with body suppressed, log-line suffix, skip-paths bypass, and `x-request-id` injection.
- **`allocation_resets_at` reset bugs (PR #148)** — two adjacent issues that would have compounded into the search-cap reset cycle:
  1. **30-day drift** — every `now + timedelta(days=30)` site accumulated ~5 days/year of error vs Apple's calendar-month billing. Replaced with `relativedelta(months=1)` for the local fallback. Apple-aware paths (the webhook handler) now anchor `allocation_resets_at` to `expiresDate` from the signed transaction — Apple's value bakes in calendar/end-of-month edge cases (Jan 31 → Feb 28 → Mar 31, leap years) and stays in sync with what Apple actually charges.
  2. **DID_RENEW same-tier no-op** — the Apple webhook historically early-returned on a renewal where the user's tier didn't change (i.e., normal monthly Plus → Plus renewal), leaving `monthly_used_usd` accumulating across renewals indefinitely and `allocation_resets_at` frozen on the original subscription date. A subscriber on a stable tier would hit their cost cap permanently after one billing cycle. Replaced the early-return with `_renew_same_tier` which zeros `monthly_used_usd` + `searches_used` and advances `allocation_resets_at` to Apple's `expiresDate`.
  3. **Lazy-reset safety net** — `usage_tracker.check_quota` now calls `lazy_reset_if_due` on every read. Atomic WHERE-guarded UPDATE that catches missed/delayed Apple webhooks AND Free users (no Apple webhook path). Multi-month inactivity gaps preserve the original day-of-month anchor by always computing from the original stale date rather than chaining single-month deltas (which loses the anchor after the first end-of-month snap-back).
  - New `app/services/allocation_reset.py` centralizes `compute_next_reset` / `roll_forward_past` / `lazy_reset_if_due`. 20 new tests covering Apple `expiresDate` override, calendar edge cases, multi-month gaps with anchor preservation, lazy-reset due/not-due/race-loss/null states, and the same-tier-renewal fix end-to-end.
  - `python-dateutil` promoted to an explicit dependency.
